## What is Booting?
\- Booting is the process that happens from the moment you press the power button on a computer to the moment the operating system is fully loaded and ready for you to use. The word "boot" comes from the phrase "pulling yourself up by your bootstraps" — meaning the computer brings itself to life using a small set of instructions that load progressively bigger and more complex software until the full OS is running.

\- From a cybersecurity perspective, the boot process is critically important because it is the foundation of everything. If an attacker can compromise the boot process, they can control the entire system — before the OS, before antivirus, before any security software even starts. This is why concepts like rootkits, bootkits, Secure Boot, and TPM exist.

---
## The Two Firmware Standards: BIOS and UEFI

\- Before we walk through the boot steps, you need to understand that there are two types of firmware that start the boot process. **Firmware** is low-level software permanently stored on a chip on the motherboard. It is the very first code that runs when power is applied.

\- **BIOS (Basic Input/Output System)** is the older standard, developed in the 1970s–80s. It is a simple 16-bit program stored on a ROM/flash chip. BIOS has significant limitations: it can only read drives up to 2.1 TB, it uses a partition scheme called **MBR** (Master Boot Record), it runs in a primitive processor mode called **Real Mode**, and its interface is a basic text menu. Most modern systems no longer use legacy BIOS, but you must understand it because legacy malware, older systems in corporate environments, and CTF challenges still involve it constantly.

\- **UEFI (Unified Extensible Firmware Interface)** is the modern replacement for BIOS. It is a full **32-bit or 64**-bit mini-operating system stored on firmware. UEFI can handle drives larger than 2.1 TB, uses a partition scheme called **GPT** (GUID Partition Table), supports a feature called **Secure Boot** (which we will cover), has a graphical interface with mouse support, can load network drivers before the OS starts, and can even run small applications. Every modern computer (post ~2012) uses UEFI, though most still have a "Legacy BIOS compatibility mode" called CSM (Compatibility Support Module).

## The Boot Process — Step by Step

### Step 1: Power On and POST (Power-On Self-Test)

\- The instant you press the power button, the **power supply unit** (PSU) begins delivering electricity to all components. Once stable voltage is achieved, the PSU sends a signal called **Power Good** to the motherboard.

\- The CPU then wakes up and immediately begins executing code from a hardcoded memory address — on x86 systems this address is `0xFFFFFFF0`, which is a location in the firmware chip (BIOS/UEFI). This is how the CPU always knows where to start, regardless of what OS is installed.

\- The first thing the firmware does is run **POST**, which is a diagnostic routine that checks whether essential hardware is present and functioning. POST checks the CPU, RAM (counts and tests memory), GPU (graphics card), keyboard, storage drives, and other connected hardware. If POST fails, the system cannot continue booting. Since there may be no display yet, errors are communicated via **beep codes** (a series of beeps with different patterns indicating different failures) or via POST LED codes on modern motherboards.

\- If POST passes successfully, the firmware knows the hardware is in a working state and moves to the next step.
### Step 2: Firmware Initialization and Hardware Configuration

\- After POST, the firmware initializes all hardware — it sets up the CPU, configures RAM timings, detects all storage devices (hard drives, SSDs, USB drives, optical drives), initializes PCI/PCIe devices, and builds a map of all hardware resources (IRQs, I/O ports, memory addresses). This is what allows the OS to later communicate with hardware — the firmware creates a standardized interface (through tables and data structures) that the OS reads.

\- In UEFI systems, this phase is more sophisticated. UEFI runs through its own internal phases called **SEC (Security)**, **PEI (Pre-EFI Initialization)**, and **DXE (Driver Execution Environment)**, during which it loads firmware drivers for each piece of hardware.

### Step 3: Boot Device Selection

\- The firmware now looks at the **boot order** — a list configured in the BIOS/UEFI settings that tells the system which device to try booting from first. A typical boot order might be: USB Drive → DVD Drive → Hard Drive. The firmware checks each device in order until it finds one that is bootable.

\- **In BIOS/MBR systems:** The firmware reads the very **first 512 bytes** of the storage device. This first sector is called the **Master Boot Record (MBR)**. The firmware checks if the last 2 bytes of this sector contain the signature `0x55AA`. If they do, the device is considered bootable. The MBR contains a tiny program (the **bootloader**, only 446 bytes) and a partition table (up to 4 primary partitions).

\- **In UEFI/GPT systems:** UEFI does not rely on the MBR mechanism. Instead, it looks for a special partition on the drive called the **EFI System Partition (ESP)**. The ESP is formatted with a FAT32 filesystem and contains **bootloader** files with `.efi` extensions (for example, `\EFI\BOOT\BOOTX64.EFI` on a Windows system or `\EFI\ubuntu\grubx64.efi` on a Linux system). UEFI can read this partition directly without needing a separate MBR bootloader, because UEFI itself has FAT32 drivers built in.

### Step 4: The Bootloader

\- The **bootloader** is a small program whose only job is to load the operating system kernel into memory. It is too small to be the OS itself — it is just an intermediary.

\- **In Windows (UEFI):** The bootloader is called **Windows Boot Manager** (`bootmgfw.efi`). It reads a configuration file called **BCD (Boot Configuration Data)** which contains entries for all installed Windows versions, and then loads `winload.efi`, which is responsible for loading the Windows kernel.

\- **In Windows (Legacy BIOS/MBR):** The 446-byte MBR bootloader simply finds and jumps to the **Volume Boot Record (VBR)** — a bootloader stored at the start of the active partition — which then loads `bootmgr` (Windows Boot Manager), which reads BCD and loads `winload.exe`.

\- **In Linux:** The most common bootloader is **GRUB2 (Grand Unified Bootloader version 2)**. GRUB2 shows a menu allowing you to choose between installed operating systems or kernels. Once you select one, GRUB2 loads the Linux kernel (a file typically named `vmlinuz`) and an **initrd/initramfs** (initial RAM disk — a temporary filesystem that contains tools needed to mount the real root filesystem).

\- The key point is that the bootloader is a critical attack surface. Attackers can replace or modify the bootloader to create **bootkits** — malware that persists at this level and loads before any OS-level security.

### Step 5: Loading the OS Kernel

\- Once the bootloader hands off control, the OS kernel loads into memory. The kernel is the core of the operating system — it manages CPU, RAM, I/O devices, processes, and security policies.

**In Windows:** `winload.efi` (or `winload.exe` in legacy mode) loads the Windows kernel `ntoskrnl.exe` and the Hardware Abstraction Layer `hal.dll` into memory. It also loads essential kernel-mode drivers. Then the kernel initializes, detects hardware (using the tables set up by UEFI/BIOS), starts memory management, and begins the process subsystem.

**In Linux:** The kernel file (`vmlinuz`) decompresses itself into memory. It then mounts the `initramfs` (a temporary root filesystem baked into RAM), runs a script inside it to load necessary drivers (for example, the driver for your specific type of storage controller so it can access the real drive), and then switches to the real root filesystem on disk. After that, the kernel starts the first process.


### Step 6: The First Process (Init System)

The kernel itself does not start user applications. Instead, after finishing its own initialization, it launches a single first process, and everything else on the system descends from this process.

**In Linux:** This first process is the **init system**. On modern Linux distributions (Ubuntu, Fedora, Kali Linux, Debian, etc.), this is **systemd** (PID 1). systemd reads its configuration and starts all the services the system needs — networking, display server, login manager, scheduled tasks, etc. Older Linux systems used **SysVinit** or **Upstart** instead. The concept of **runlevels** (SysVinit) or **targets** (systemd) defines which set of services to start — for example, a "multi-user with networking but no GUI" mode versus a "full graphical desktop" mode.

**In Windows:** The kernel starts `smss.exe` (Session Manager Subsystem), which starts `csrss.exe` (Client/Server Runtime Subsystem) and `wininit.exe`. `wininit.exe` starts `services.exe` (the Service Control Manager, which starts all Windows services), `lsass.exe` (Local Security Authority Subsystem Service — handles authentication and security policy), and `lsaiso.exe` (on systems with Credential Guard). Eventually `winlogon.exe` starts the login screen. After login, `explorer.exe` provides the desktop environment.

### Step 7: User Login and Full OS Operation

The login screen is presented by the OS. After successful authentication, the user profile is loaded, the desktop environment starts, and the system is fully operational.

## MBR vs GPT — A Deeper Look (Important for Hacking Context)

**MBR (Master Boot Record)** occupies the first 512 bytes of a disk. The layout is: 446 bytes of bootloader code + 64 bytes of partition table (4 entries × 16 bytes each) + 2 bytes signature (0x55AA). MBR supports a maximum disk size of 2.1 TB and a maximum of 4 primary partitions (you can work around this using one "extended partition" containing "logical partitions"). MBR has no integrity protection — anyone with disk write access can overwrite it. This is why MBR-based bootkits were so effective and common.

**GPT (GUID Partition Table)** is the modern standard used with UEFI. GPT stores a backup of the partition table at the end of the disk. It supports up to 128 partitions by default and handles disks up to 9.4 ZB (zettabytes). Each partition has a globally unique identifier (GUID). GPT also has a **Protective MBR** in the first sector — this is not a real MBR but a dummy that tells old MBR-only tools "this disk is in use, do not touch it."

## Secure Boot — What It Is and Why It Matters

**Secure Boot** is a UEFI feature designed to ensure that only trusted, cryptographically signed software can run during the boot process. Here is how it works:

The UEFI firmware contains a database of trusted cryptographic keys — specifically, a **Platform Key (PK)**, a **Key Exchange Key (KEK)**, and a **Signature Database (db)**. When Secure Boot is enabled, UEFI checks the digital signature of every bootloader and kernel it is about to execute against the keys in its database. If the signature is valid (signed by Microsoft, for Windows, or by a trusted CA for Linux), execution proceeds. If the signature is missing or invalid, UEFI refuses to boot and shows an error.

This means an attacker cannot simply replace `bootmgfw.efi` with a malicious bootloader — UEFI will reject it because it is not signed with a trusted key.

Secure Boot can be disabled in UEFI settings, which is sometimes necessary for installing certain Linux distributions or running older software. In penetration testing, finding a system with Secure Boot disabled is a significant finding, as it removes a major layer of boot-time protection.

## TPM (Trusted Platform Module) — Brief Overview

**TPM** is a dedicated hardware chip (or firmware implementation) that performs cryptographic functions and stores secrets securely. In the context of booting, TPM is used alongside features like **BitLocker** (Windows drive encryption) and **Measured Boot**. During Measured Boot, each stage of the boot process (firmware → bootloader → kernel → drivers) takes a cryptographic measurement (hash) of the next component before executing it and stores these hashes in TPM registers called **PCRs (Platform Configuration Registers)**. If anything in the boot chain is tampered with, the PCR values change, which BitLocker detects and responds to by refusing to release the encryption key — effectively locking the drive.

## Cybersecurity Relevance — Attack Surfaces in the Boot Process

Understanding the boot process is essential because each stage is a potential attack surface:

**MBR/VBR Attacks (Bootkits):** Malware like TDL4 (Alureon), Rovnix, and Necurs overwrote or infected the MBR or VBR so they loaded before Windows. Because they ran before the OS, they were invisible to OS-level antivirus and could intercept OS function calls (rootkit behavior). Tools like `bootrec /fixmbr` (Windows Recovery) or `dd` on Linux can restore the MBR, but sophisticated bootkits hid themselves even from these.

**UEFI/Firmware Implants:** Nation-state malware like **LoJax** (discovered 2018) and **MosaicRegressor** modified UEFI firmware itself — the code on the chip — so that even wiping the drive and reinstalling the OS did not remove the malware. This is the deepest persistence possible. The only defense is Secure Boot, TPM, and keeping firmware updated.

**Evil Maid Attacks:** If an attacker has physical access to an unattended computer ("evil maid" in a hotel room scenario), they can boot from a USB drive, bypass disk encryption if Secure Boot is off, modify the bootloader to capture passwords typed at login, and restore everything so the victim never knows. This is why Full Disk Encryption + Secure Boot + BIOS password + TPM together are recommended.

**Cold Boot Attack:** RAM retains data for a short period (seconds to minutes) after power loss, especially if cooled. An attacker can power off a running computer, quickly boot it from USB, and dump RAM contents to recover encryption keys that were stored there. This is why **BitLocker with TPM + PIN** is more secure than TPM alone.

**Boot Order / USB Boot Attacks:** If an attacker can change the boot order in UEFI to boot from a USB drive, they can boot into a live OS (like Kali Linux), mount the victim's drive, and access or modify files. The countermeasure is setting a UEFI/BIOS password and disabling USB booting.

**Bootloader Credential Theft:** An attacker with prior OS access or physical access can replace a legitimate bootloader with a modified one that shows the normal login screen but logs every password entered — a form of keylogger at the bootloader level.

## Key Files Reference — Windows Boot Components

|File|Location|Role|
|---|---|---|
|`bootmgfw.efi`|ESP `\EFI\Microsoft\Boot\`|Windows Boot Manager (UEFI)|
|`bootmgr`|Root of system partition|Windows Boot Manager (Legacy)|
|`BCD`|`\Boot\BCD`|Boot Configuration Data store|
|`winload.efi` / `winload.exe`|`\Windows\System32\`|Loads the Windows kernel|
|`ntoskrnl.exe`|`\Windows\System32\`|Windows kernel|
|`hal.dll`|`\Windows\System32\`|Hardware Abstraction Layer|
|`smss.exe`|`\Windows\System32\`|Session Manager (first user-mode process)|
|`lsass.exe`|`\Windows\System32\`|Authentication and security (huge hacking target)|

## Summary: The Complete Boot Sequence at a Glance

Power On → PSU sends Power Good → CPU starts executing firmware code → POST (hardware check) → Firmware initializes hardware → Firmware checks boot order → Firmware finds bootable device (MBR signature or UEFI ESP) → Bootloader runs (GRUB2 or Windows Boot Manager) → Bootloader loads OS kernel → Kernel initializes, loads drivers → Init system (systemd / Windows services) starts → Login screen → User authenticated → Desktop loaded → System ready.
