# SSH Server ‚Äî Setup & Configuration Guide

---
## 1. Theory

### What is SSH?

- **SSH (Secure Shell)** is a cryptographic network protocol for securely accessing and managing a remote machine over an unsecured network.
- Default port: **22 (TCP)**
- Replaces older insecure protocols ‚Äî Telnet, rlogin, rsh ‚Äî which sent data in plaintext.

### What SSH Does

- **Remote encrypted connections** ‚Äî all data between client and server is encrypted. An interceptor sees only ciphertext (random noise).
- **Authentication** ‚Äî verifies identity via password or cryptographic key pair.
- **Tunneling / Port Forwarding** ‚Äî wraps packets with new headers to move them across networks or firewalls they otherwise couldn't traverse.
- **File transfer** ‚Äî `scp` and `sftp` are built on top of SSH.

### How SSH Authentication Works

1. Client connects to server on port 22.
2. Server sends its **public host key** ‚Äî client checks it against `~/.ssh/known_hosts`.
3. Authentication proceeds via:
    - **Password** ‚Äî user provides their system password.
    - **Public Key** ‚Äî client proves identity using a private key matched to a public key stored in `~/.ssh/authorized_keys` on the server. More secure; no password needed.
4. An encrypted session is established.

### Key Files

|File|Location|Purpose|
|---|---|---|
|`sshd_config`|`/etc/ssh/sshd_config`|Server configuration file|
|`authorized_keys`|`~/.ssh/authorized_keys`|Public keys of clients allowed to log in|
|`known_hosts`|`~/.ssh/known_hosts`|Client-side record of trusted server host keys|
|Host keys|`/etc/ssh/ssh_host_*`|Server's own identity keys (auto-generated on install)|

|Distro Family|Distros|Package|Service Name|
|---|---|---|---|

## 3. Installing OpenSSH Server

### Debian / Ubuntu / Kali / Mint / ParrotOS

Update package list:

```bash
sudo apt update
```

Install OpenSSH server:

```bash
sudo apt install openssh-server -y
```

Enable SSH to start at boot:

```bash
sudo systemctl enable ssh
```

Start the SSH service:

```bash
sudo systemctl start ssh
```

Check service status:

```bash
sudo systemctl status ssh
```

---

## 4. Firewall Configuration

### Firewalld ‚Äî CentOS / RHEL / Fedora / AlmaLinux / Rocky

Permanently allow SSH through the firewall:

```bash
sudo firewall-cmd --permanent --add-service=ssh
```

Reload firewall to apply changes:

```bash
sudo firewall-cmd --reload
```

Verify SSH is listed:

```bash
sudo firewall-cmd --list-all
```

---

### UFW ‚Äî Ubuntu / Debian / Kali / Mint

Allow SSH (port 22):

```bash
sudo ufw allow ssh
```

Enable UFW if not already active:

```bash
sudo ufw enable
```

Check UFW status:

```bash
sudo ufw status
```

---

### iptables ‚Äî Any distro (manual / legacy)

Allow incoming connections on port 22:

```bash
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

## 5. Server Configuration (`sshd_config`)

The main SSH server config file: `/etc/ssh/sshd_config`

Open with nano:

```bash
sudo nano /etc/ssh/sshd_config
```

Open with vim:

```bash
sudo vim /etc/ssh/sshd_config
```

### Key Configuration Options

```bash
# Port SSH listens on (default 22)
Port 22

# Allow root login ‚Äî set to 'no' in production
PermitRootLogin yes

# Allow password-based login
PasswordAuthentication yes

# Allow public key authentication (recommended)
PubkeyAuthentication yes

# Disallow empty passwords (always keep 'no')
PermitEmptyPasswords no

# Max failed auth attempts before disconnect
MaxAuthTries 3

# Disconnect idle sessions after 300 seconds of inactivity
ClientAliveInterval 300
ClientAliveCountMax 2

# Restrict login to specific users only (optional whitelist)
# AllowUsers risabh john
```

After editing, restart SSH to apply changes.

Restart on Debian-based systems:

```bash
sudo systemctl restart ssh
```

Restart on RHEL / Arch / SUSE:

```bash
sudo systemctl restart sshd
```

Verify service is still running after restart ‚Äî Debian-based:

```bash
sudo systemctl status ssh
```

Verify service is still running after restart ‚Äî RHEL / Arch / SUSE:

```bash
sudo systemctl status sshd
```

## 6. Verify SSH is Listening on Port 22

Check using `ss`:

```bash
ss -tlnp | grep 22
```

Check using `netstat` (if installed):

```bash
netstat -tlnp | grep 22
```

Expected output will show `0.0.0.0:22` or `:::22` (IPv6).

## 7. Setting Up Public Key Authentication

More secure than passwords ‚Äî no password is ever sent over the network.

### On the Client Machine ‚Äî Generate a key pair

Generate an ed25519 key pair (recommended):

```bash
ssh-keygen -t ed25519 -C "your_label"
```

This creates:

- `~/.ssh/id_ed25519` ‚Äî private key (**never share this**)
- `~/.ssh/id_ed25519.pub` ‚Äî public key (copy to the server)

### Copy the Public Key to the Server

Using `ssh-copy-id` (easiest):

```bash
ssh-copy-id username@server-ip
```

Manual alternative (if `ssh-copy-id` is unavailable):

```bash
cat ~/.ssh/id_ed25519.pub | ssh username@server-ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

### Test Key-Based Login

```bash
ssh username@server-ip
```

You should connect without being asked for a password.

### (Optional) Disable Password Auth After Confirming Key Works

In `/etc/ssh/sshd_config`, set:

```bash
PasswordAuthentication no
```

Then restart SSH to apply.

## 8. Common Errors and Fixes

### Error: `REMOTE HOST IDENTIFICATION HAS CHANGED`

```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@ WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!        @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
```

**Cause:** Server's host key changed ‚Äî happens when OS is reinstalled, VM is redeployed with same IP, or IP was reassigned to a different machine.

Remove the old entry from known_hosts:

```bash
ssh-keygen -R 192.168.1.100
```

Then reconnect normally ‚Äî SSH will prompt to accept the new key:

```bash
ssh root@192.168.1.100
```

Manual fix ‚Äî open known_hosts and delete the offending line:

```bash
nano ~/.ssh/known_hosts
```

Find the line containing the server IP or hostname, delete it, save, then reconnect.

---

### Error: `Connection refused`

**Causes:** SSH service not running, wrong port, or firewall blocking port 22.

Check SSH service status ‚Äî RHEL-based:

```bash
sudo systemctl status sshd
```

Check SSH service status ‚Äî Debian-based:

```bash
sudo systemctl status ssh
```

Check what port SSH is actually listening on:

```bash
ss -tlnp | grep ssh
```

---

### Error: `Permission denied (publickey)`

**Cause:** Password auth is disabled on the server and no key is configured.

Enable verbose mode to see what's happening:

```bash
ssh -v user@server-ip
```

Specify a key file explicitly:

```bash
ssh -i ~/.ssh/id_ed25519 user@server-ip
```

---

### Error: `Host key verification failed`

Same cause and fix as the `REMOTE HOST IDENTIFICATION HAS CHANGED` error above.

Remove stale entry:

```bash
ssh-keygen -R server-ip
```

## 9. Viewing SSH Logs

View live SSH logs ‚Äî Debian-based:

```bash
sudo journalctl -u ssh -f
```

View live SSH logs ‚Äî RHEL / Arch / SUSE:

```bash
sudo journalctl -u sshd -f
```

Auth log ‚Äî Debian / Ubuntu / Kali:

```bash
sudo tail -f /var/log/auth.log
```

Auth log ‚Äî CentOS / RHEL:

```bash
sudo tail -f /var/log/secure
```
