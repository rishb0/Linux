# Linux Important Files, Permissions, and Pentesting Relevance

---
## Understanding Permissions First

Before reading this document, you need to understand how Linux file permissions work, because every file listed here is described with its default permission, and without knowing what that means you cannot understand whether that file is a risk or not.

Linux permissions are displayed as a 10-character string like `-rwxr-xr--`. The first character is the file type (`-` for regular file, `d` for directory, `l` for symlink). The next 9 characters are split into three groups of three: the first group is the **owner's** permissions, the second group is the **group's** permissions, and the third group is **everyone else's** (also called "other" or "world") permissions. Each group has three slots: `r` (read), `w` (write), `x` (execute). A `-` in a slot means that permission is not granted.

So `-rw-r--r--` means: it is a regular file, the owner can read and write it, the group can only read it, and everyone else on the system can also read it. This is also written numerically as `644`.

The key thing to remember for pentesting is: if the "other" group (the third triplet) has `r`, then **any user on the system can read that file**, including a low-privilege shell you just obtained via exploitation.

## Authentication and Credential Files

### /etc/passwd

- `/etc/passwd` has permissions `644`, which means it is world-readable — every user on the system can read it.
- This file stores one entry per user account, and each entry contains the username, a placeholder for the password (which is now just an `x`), the user ID (UID), the group ID (GID), a comment field (often the full name), the home directory path, and the default login shell.
- The format of each line is: `username:x:UID:GID:comment:home_dir:shell`.
- An example entry looks like: `risabh:x:1001:1001:Risabh Kumar:/home/risabh:/bin/bash`.
- The password is not actually stored here anymore — the `x` just tells the system to look in `/etc/shadow` instead, which is not world-readable.
- From a pentesting perspective, this file gives you a full list of every user account on the machine, their UIDs, their home directories, and their login shells, which helps you understand what users exist and which ones have valid login shells (accounts with `/bin/false` or `/usr/sbin/nologin` as their shell cannot log in interactively).
- Some very old or misconfigured systems still store actual password hashes directly in this file instead of using shadow passwords, and if you find a hash there you can crack it offline.

### /etc/shadow

- `/etc/shadow` has permissions `640` by default, owned by root and the group `shadow`, meaning **only root and members of the shadow group can read it** — a regular user cannot.
- This file stores the actual password hashes for every user account, along with password aging and expiry information.
- The format of each line is: `username:hashed_password:last_changed:min_age:max_age:warn_days:inactive_days:expire_date:reserved`.
- The hash field uses a format like `$6$salt$hash`, where `$6$` indicates SHA-512 hashing, `$5$` is SHA-256, `$1$` is MD5, and `$y$` is yescrypt (used in modern Debian/Ubuntu).
- A `*` or `!` in the hash field means the account is locked and cannot be logged into with a password.
- If you can read this file (via LFI, via a root shell, or via a SUID binary), you extract the hashes and run them through a tool like hashcat or john offline to crack them.

### /etc/group

- `/etc/group` has permissions `644`, so it is world-readable.
- This file defines all the groups on the system and which users belong to each group.
- The format is: `groupname:x:GID:member1,member2,member3`.
- From a pentesting perspective, this tells you which users are in the `sudo` group (and can therefore run commands as root), which users are in `docker` (which can be abused for privilege escalation), which users are in `disk` (direct access to block devices), and so on.

### /etc/gshadow

- `/etc/gshadow` has permissions `640`, readable only by root and the shadow group.
- This file is the group equivalent of `/etc/shadow` — it stores group passwords and administrator information for groups.
- It is rarely populated in modern systems but if it is, it can contain hashes.

### /etc/sudoers

- `/etc/sudoers` has permissions `440`, readable only by root, not writable by anyone directly (you must use the `visudo` command to edit it safely).
- This file defines who can run what commands as root or as other users using `sudo`.
- Misconfigurations here are one of the most common privilege escalation paths — for example, a line like `risabh ALL=(ALL) NOPASSWD: /usr/bin/python3` means that user risabh can run python3 as root without a password, which can be abused to spawn a root shell.
- The directory `/etc/sudoers.d/` contains additional sudoers configuration files that are included into the main file — always check these too.

### /etc/security/opasswd

- `/etc/security/opasswd` has permissions `600`, readable only by root.
- This file stores old password hashes for users when password history is enforced (so users cannot reuse recent passwords).
- If you can read it, it gives you additional hashes to crack.

## SSH Files

### /etc/ssh/sshd_config

- `/etc/ssh/sshd_config` has permissions `600` or `644` depending on the distribution, owned by root.
- This is the SSH server's configuration file — it controls which users can log in, whether password authentication is allowed or only key-based, which port SSH listens on, whether root login is permitted, and many other settings.
- From a pentesting perspective, if you can read this, you learn whether password-based SSH login is allowed (which tells you whether stolen credentials can be used to log in remotely), whether root can log in directly via SSH, and what port SSH is running on if it was changed from the default 22.

### ~/.ssh/authorized_keys

- `~/.ssh/authorized_keys` has permissions `600`, readable only by the owning user and root.
- This file inside a user's home directory contains the list of public keys that are allowed to log in as that user via SSH without a password.
- If you can **write** to this file (for example via a file write vulnerability or via a misconfigured web shell), you can add your own public key and then SSH in as that user without needing a password.

### ~/.ssh/id_rsa (and id_ecdsa, id_ed25519)

- `~/.ssh/id_rsa` has permissions `600`, readable only by the owning user and root.
- This is the user's **private** SSH key — if you can read this file, you can use it to log in to any remote server that has the corresponding public key in its `authorized_keys`.
- This is one of the highest-value files to find during a pentest because a single private key often gives access to multiple servers.
- The file starts with `-----BEGIN OPENSSH PRIVATE KEY-----` or `-----BEGIN RSA PRIVATE KEY-----`.
- If the key is passphrase-protected, you still need to crack that passphrase, but many private keys have no passphrase at all.

### ~/.ssh/known_hosts

- `~/.ssh/known_hosts` has permissions `644` or `600`.
- This file records the public keys of every remote server that user has previously connected to via SSH, so that SSH can warn the user if the server's key changes unexpectedly.
- From a pentesting perspective, this file reveals what other machines this user has connected to, giving you a map of other targets in the network.

## System Configuration Files

### /etc/hosts

- `/etc/hosts` has permissions `644`, world-readable.
- This file maps hostnames to IP addresses locally, before DNS is consulted.
- It often reveals internal hostnames and IP addresses that are not exposed via public DNS, showing you internal infrastructure like `192.168.1.50 database.internal` or `10.0.0.20 jenkins.company.local`.

### /etc/resolv.conf

- `/etc/resolv.conf` has permissions `644`, world-readable.
- This file configures which DNS servers the machine uses for name resolution.
- It can reveal internal DNS server addresses, which are often also domain controllers in Windows-integrated environments.

### /etc/fstab

- `/etc/fstab` has permissions `644`, world-readable.
- This file defines all the filesystems that are automatically mounted at boot, including network shares (NFS, SMB/CIFS), disk partitions, and special filesystems.
- It often reveals NFS share paths, SMB credentials embedded in mount options (in misconfigured systems), and the layout of the disk structure.

### /etc/hostname

- `/etc/hostname` has permissions `644`, world-readable.
- This file simply contains the machine's hostname, which helps you understand what machine you are on and what role it plays in the network.

### /etc/os-release

- `/etc/os-release` has permissions `644`, world-readable.
- This file contains the full details of the Linux distribution and version — for example, `Ubuntu 22.04.3 LTS` — which is critical for finding version-specific exploits and knowing which default paths and package locations to use.

### /etc/environment and /etc/profile

- `/etc/environment` has permissions `644`, world-readable, and it contains system-wide environment variables that apply to all users and all sessions.
- `/etc/profile` has permissions `644`, world-readable, and it is a shell script that runs for every user at login to set up their shell environment, and it can contain variable definitions, path additions, or other configurations that reveal how the system is set up.
- These files sometimes contain hardcoded credentials, API keys, database URLs, or application tokens placed there by administrators for convenience.

### /etc/crontab and /etc/cron.d/

- `/etc/crontab` has permissions `644`, world-readable.
- This file defines scheduled tasks that run as root or as specific users at defined intervals.
- The directory `/etc/cron.d/` contains additional cron job files with the same format.
- The directories `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, and `/etc/cron.monthly/` contain scripts that run at those intervals.
- From a pentesting perspective, cron jobs are one of the most common privilege escalation vectors — if a cron job runs as root and calls a script or binary that you can write to, you can modify that script to execute your own commands as root.
- Even if the cron job file itself is not writable, if it calls a script in a world-writable directory, you can replace that script.

### /var/spool/cron/crontabs/

- `/var/spool/cron/crontabs/` has permissions `1730` (sticky, readable by root and crontab group), and individual files inside are `600`, readable only by root.
- These are per-user crontab files — each user who has set up cron jobs has a file here named after their username.
- If you can read these, you learn about scheduled tasks running under specific user accounts.

## Web Server Configuration Files

### /etc/apache2/ (Debian/Ubuntu) or /etc/httpd/ (RHEL/CentOS)

- The main Apache configuration file is `/etc/apache2/apache2.conf` or `/etc/httpd/conf/httpd.conf`, with permissions `644`.
- The directory `/etc/apache2/sites-available/` (or `/etc/apache2/conf.d/`) contains virtual host configuration files that define what websites are hosted, where their document roots are, and what PHP or other settings are applied.
- These files reveal the exact document root paths of web applications (where to look for files), enabled PHP settings like `allow_url_include` (which enables RFI), the server's running user (often `www-data`), and any custom log file locations.

### /etc/nginx/nginx.conf and /etc/nginx/sites-available/

- `/etc/nginx/nginx.conf` is the main Nginx configuration, with permissions `644`.
- `/etc/nginx/sites-available/` contains per-site configuration files.
- These reveal document roots, proxy pass rules (if Nginx is forwarding to a backend app), and logging configuration.

### /var/www/html/

- `/var/www/html/` is the default Apache web root on Debian/Ubuntu systems, and the web server user (`www-data`) has read access.
- This is where web application files live — PHP files, configuration files, `.env` files, database connection files like `wp-config.php` (WordPress), `config.php`, `database.php`, and so on.
- These files often contain plaintext database credentials (hostname, username, password, database name) because the application needs them to connect to the database.

### Common application config files to look for in web roots

- `wp-config.php` — WordPress database credentials.
- `.env` — Laravel, Django, Node.js, and many modern frameworks store database URLs, API keys, secret keys, and SMTP credentials here.
- `config.php`, `configuration.php`, `settings.php` — generic config files in custom applications.
- `database.yml` — Ruby on Rails database configuration with credentials.
- `application.properties` or `application.yml` — Spring Boot (Java) application configuration.
- `.htpasswd` — Apache's HTTP basic auth password file, which contains usernames and hashed passwords for password-protected directories.

## /proc Virtual Filesystem

The `/proc` filesystem is not a real filesystem on disk — it is a virtual interface that the Linux kernel exposes in memory to allow you to read information about running processes, the kernel, network state, and the current process itself. All of these paths exist only while the system is running, but they can be read like regular files.

### /proc/self/environ

- `/proc/self/environ` has permissions `400`, readable only by the owner of the process (or root).
- This file contains all the environment variables of the current process, in a null-byte-separated format.
- When a web server process reads this file, it exposes variables like `HTTP_USER_AGENT`, `HTTP_HOST`, `DOCUMENT_ROOT`, `SCRIPT_FILENAME`, and others that the web server received from the HTTP request.
- This is directly exploitable via LFI for shell access — you can inject PHP code into your `User-Agent` header, and then include `/proc/self/environ` via LFI, which causes the PHP interpreter to execute your injected code because it is now part of a file being included.

### /proc/self/fd/

- `/proc/self/fd/` is a directory containing symbolic links to every file descriptor that the current process has open — file descriptor 0 is stdin, 1 is stdout, 2 is stderr, and higher numbers are files the process opened.
- When a PHP web server processes a request, it typically has file descriptors open to the access log, error log, and potentially other files.
- By including `/proc/self/fd/10` (or whatever number corresponds to the log file), you can include the log file and trigger code execution if you have already poisoned it with PHP code.
- You find the right file descriptor number by brute-forcing — trying `/proc/self/fd/0` through `/proc/self/fd/30` or higher.

### /proc/version

- `/proc/version` is world-readable and contains the kernel version string, for example: `Linux version 5.15.0-91-generic (buildd@...) (gcc version 11.4.0 ...) #101-Ubuntu SMP`.
- This tells you the exact kernel version, which you can look up to find known kernel exploits for privilege escalation.

### /proc/cmdline

- `/proc/cmdline` is world-readable and contains the parameters passed to the kernel at boot time.
- This can reveal details about the boot configuration, encrypted partition setups, or custom kernel parameters.

### /proc/net/tcp and /proc/net/tcp6

- `/proc/net/tcp` is world-readable and contains a table of all active TCP connections on the system, with local and remote addresses and ports encoded in hexadecimal.
- By decoding these hex values, you can see what internal services are listening on what ports — including services that are only bound to `127.0.0.1` (localhost) and are not visible from the outside, like internal databases, admin panels, or application APIs.

### /proc/net/arp

- `/proc/net/arp` is world-readable and contains the ARP table — a mapping of IP addresses to MAC addresses of machines on the local network.
- This reveals what other machines are on the same local network as the server, giving you targets for lateral movement.

## Log Files

Log files are critical for LFI exploitation because many of them are readable by the web server process, and they record content from HTTP requests (like headers and URLs), which you control. This means you can inject PHP code into a log file and then include that log file via LFI to execute the code.

### /var/log/apache2/access.log (or /var/log/httpd/access_log on RHEL)

- Permissions are typically `640`, owned by `root` and readable by the `adm` or `www-data` group — the web server process can often read this.
- This log records every HTTP request made to the server: the client IP, timestamp, HTTP method, URL path, response code, response size, referrer, and User-Agent.
- The User-Agent field is recorded verbatim and without sanitization, so if you send a request with a PHP shell as your User-Agent (like `<?php system($_GET['cmd']); ?>`), that exact string gets written into the log file, and you can then include the log via LFI to execute it.

### /var/log/apache2/error.log (or /var/log/httpd/error_log)

- Permissions are typically `640`, similar to the access log.
- This log records PHP errors, 404 errors, 500 errors, and other server-side errors.
- You can trigger a log entry containing PHP code by requesting a URL that includes your payload — for example, `GET /<?php system($_GET['cmd']); ?>` — which Apache records in the error log when it cannot find the file.

### /var/log/nginx/access.log and /var/log/nginx/error.log

- These are the Nginx equivalents of the Apache logs, with the same poisoning potential.
- Permissions are typically `640`, owned by root and readable by the `adm` or `www-data` group.

### /var/log/auth.log (Debian/Ubuntu) or /var/log/secure (RHEL/CentOS)

- Permissions are `640`, readable by root and the `adm` group — but the web server user (`www-data`) typically cannot read this directly.
- This log records all authentication events: SSH login attempts, sudo usage, PAM authentication, su attempts, and failed logins.
- Importantly, it records the **username** that was attempted during SSH login — so if you connect via SSH and type a PHP shell as the username (like `<?php system($_GET['cmd']); ?>`), that string gets written into this log.
- If the web server can read this file (sometimes possible on misconfigured systems), you can execute the code via LFI.

### /var/log/mail.log and /var/mail/www-data

- `/var/log/mail.log` records mail server activity, and `/var/mail/` contains the mailbox files for system users.
- If the system has a mail server and you can send a mail to a system user (like `www-data@localhost`) with PHP code in the body, the mail is stored in `/var/mail/www-data`.
- The web server process often owns that file and can read it, so including it via LFI executes your code.

### /var/log/vsftpd.log

- Permissions are typically `600` or `640`.
- This log records FTP connections, and it logs the username used to log in — meaning if you connect to an FTP service and use a PHP shell as your username, it gets written into this log.

### /var/log/syslog (Debian/Ubuntu) or /var/log/messages (RHEL/CentOS)

- Permissions are `640`, owned by root and the `adm` group.
- This is the general system log — it records kernel messages, daemon activity, cron job execution, and various system events.
- Less useful for log poisoning but very useful for understanding what the system is doing and what services are running.

### /var/log/kern.log

- Permissions are `640`.
- This contains kernel-level log messages — hardware events, driver activity, OOM killer events, and firewall rule hits (iptables/nftables log rules).
- Useful for understanding the kernel environment and whether a firewall is active.

### /var/log/dpkg.log (Debian/Ubuntu) or /var/log/yum.log (RHEL)

- Permissions are `644`, world-readable.
- These log all package installations, removals, and upgrades with timestamps.
- From a pentesting perspective, this tells you what software is installed, what versions, and when it was installed — helping you identify outdated and potentially vulnerable packages.

### /var/log/cron.log or /var/log/cron (RHEL)

- Permissions are `640`.
- This logs every execution of cron jobs — what command ran, at what time, under what user.
- Combined with the crontab files, this helps you verify whether a cron job is actually running and confirm your privilege escalation attempt worked.

### /var/log/wtmp and /var/log/btmp

- `/var/log/wtmp` records every successful login and logout — it is a binary file read with the `last` command.
- `/var/log/btmp` records every failed login attempt — read with `lastb`.
- Permissions: `wtmp` is `664` (world-readable), `btmp` is `660` (readable only by root and utmp group).
- These tell you who has logged into the system and from where.

### /var/log/lastlog

- Permissions are `644`, world-readable.
- This binary file records the last login time and source for every user account, readable with the `lastlog` command.

## History and Shell Configuration Files

### ~/.bash_history

- `~/.bash_history` has permissions `600`, readable only by the owning user and root.
- This file stores the history of commands that user has typed in bash — and administrators frequently type passwords, database connection strings, API keys, and curl commands with credentials directly into the terminal.
- This is one of the first files to check after getting a shell as any user.

### ~/.bashrc and ~/.bash_profile and ~/.profile

- These files have permissions `644` or `600` and run when the user opens a shell.
- They often contain environment variable definitions like `export DB_PASS=secret123` or `export AWS_SECRET_KEY=...` added by the user for convenience.

### /root/.bash_history

- Same as above but for the root user — only readable by root, but if you escalate privileges to root this is the first thing to check.

## Password and Credential Files Summary

To directly answer the question of "what are more password files like passwd" — here is every file on a Linux system that contains passwords or password hashes:

- `/etc/passwd` — user account list (contains `x` placeholder now, but on old systems contains actual hashes).
- `/etc/shadow` — actual password hashes for all users (requires root or shadow group to read).
- `/etc/gshadow` — group password hashes (requires root or shadow group).
- `/etc/security/opasswd` — old/previous password hashes for history enforcement (root only).
- `~/.ssh/id_rsa` — SSH private key (effectively a credential file since it grants access without a password).
- `~/.bash_history` — may contain plaintext passwords typed in commands.
- `/var/www/html/wp-config.php` or any app config file — plaintext database passwords.
- `.env` files throughout web roots — plaintext credentials for databases, APIs, mail servers.
- `.htpasswd` files — hashed passwords for Apache HTTP Basic Auth.
- `/etc/pam.d/` configuration files — these do not contain passwords but define how authentication is performed and can reveal if non-standard auth methods are in use.
- `/etc/krb5.keytab` — Kerberos keytab file, permissions `600`, contains service account keys for Kerberos authentication (highly sensitive in domain environments).

## Permissions That Are Always Dangerous

Regardless of which specific file you encounter, these permission patterns are always a concern during a pentest:

- A file that should be private (like a config file with credentials) but has world-readable permissions (`o+r`) — any user or process on the system can read it.
- A file or directory that is world-writable (`o+w`) — any user can modify or replace it, which is a direct privilege escalation path if that file is executed by root.
- A binary with the SUID bit set (`-rwsr-xr-x`) — when any user runs this binary, it executes with the permissions of the file owner (usually root), which can be abused to escalate privileges.
- A directory with sticky bit not set (`+t`) in a shared writable directory — without sticky bit, users can delete each other's files, enabling certain attacks.
