# Linux Directory Structure

Linux follows the **Filesystem Hierarchy Standard (FHS)** вЂ” a consistent layout used across all major Linux distributions. Everything starts from a single root `/` and branches into subdirectories.

## Directory Reference

### `/` вЂ” Root Directory

- Top-level directory; everything branches from here.
- Only the **root user** has full unrestricted access.

---

### `/bin` вЂ” Essential User Binaries

- Core user commands needed for booting, repairing, or running the system in single-user mode.
- Examples: `ls`, `cat`, `cp`, `mkdir`, `sh`, `bash`
- рџ”ґ **Pentesting:** `sh` and `bash` here are used for scripting and spawning shells.

---
### `/sbin` вЂ” System Binaries

- Essential system administration commands вЂ” intended for the root user.
- Examples: `iptables`, `reboot`, `ifconfig`, `fdisk`
- рџ”ґ **Pentesting:** `iptables` can be inspected for firewall rules or manipulated for firewall evasion.

---
### `/etc` вЂ” Configuration Files

- System-wide configuration files for all services and applications.

|File|Purpose|
|---|---|
|`/etc/passwd`|User account info (username, UID, shell, etc.)|
|`/etc/shadow`|Hashed passwords вЂ” readable by root only|
|`/etc/ssh/sshd_config`|SSH server configuration|
|`/etc/hosts`|Local DNS hostname-to-IP mappings|
|`/etc/crontab`|Scheduled tasks (check for misconfigs)|

- рџ”ґ **Pentesting:** Prime target вЂ” misconfigurations, exposed credentials, weak permissions on shadow, SUID abuse via cron.

---
### `/dev` вЂ” Device Files

- Contains device nodes representing hardware components.

| File        | Purpose                                        |
| ----------- | ---------------------------------------------- |
| `/dev/null` | "Black hole" вЂ” discards all data written to it |
| `/dev/zero` | Outputs a stream of null bytes                 |
| `/dev/sda`  | First hard disk                                |
| `/dev/tty`  | Current terminal                               |

- рџ”ґ **Pentesting:** `/dev/input` can be exploited for keylogging on physical or interactive sessions.

---
### `/proc` вЂ” Process Information

- A **virtual filesystem** вЂ” not real files on disk; generated by the kernel at runtime.
- Contains live information about running processes and system state.

|Path|Purpose|
|---|---|
|`/proc/cpuinfo`|CPU details|
|`/proc/meminfo`|Memory usage|
|`/proc/[PID]/cmdline`|Command-line args of a running process|
|`/proc/[PID]/environ`|Environment variables of a process|
|`/proc/net/tcp`|Active TCP connections|

- рџ”ґ **Pentesting:** Process cmdline and environment variables can leak passwords, tokens, or connection strings passed at runtime.

---
### `/var` вЂ” Variable Data

- Files expected to grow over time вЂ” logs, caches, mail, web content.

| Path                | Purpose                                      |
| ------------------- | -------------------------------------------- |
| `/var/log/auth.log` | Auth events, SSH logins, sudo вЂ” Debian-based |
| `/var/log/secure`   | Same as above вЂ” RHEL-based systems           |
| `/var/log/syslog`   | General system messages                      |
| `/var/www/html`     | Default web root for Apache/Nginx            |
| `/var/mail`         | User mailboxes                               |

- рџ”ґ **Pentesting:** `auth.log` reveals brute-force attempts and successful logins. `/var/www` is the target for web app testing and webshell uploads.

---
### `/home` вЂ” User Home Directories

- Each regular user gets a personal directory: `/home/<username>/`

|Path|Purpose|
|---|---|
|`/home/user/.ssh/`|SSH keys and `authorized_keys`|
|`/home/user/.bash_history`|Command history вЂ” often contains plaintext passwords|
|`/home/user/.bashrc`|User shell config|

- рџ”ґ **Pentesting:** Weak permissions expose private SSH keys; history files leak credentials; `authorized_keys` can be backdoored for persistent access.

---
### `/root` вЂ” Root User's Home Directory

- Home directory exclusively for the **root user**.
- Contains root's SSH keys, scripts, and personal configs.
- рџ”ґ **Pentesting:** The ultimate target during privilege escalation вЂ” full system compromise.

---
### `/tmp` вЂ” Temporary Files

- **World-writable** вЂ” any user or process can write here.
- Cleared on reboot.
- рџ”ґ **Pentesting:** Standard location for uploading payloads, compiling exploits, or staging files during post-exploitation. Always check for sensitive files left behind.

---
### `/usr` вЂ” User Utilities and Applications

- Secondary hierarchy for non-essential binaries and shared resources.

| Subdirectory | Purpose                                                              |
| ------------ | -------------------------------------------------------------------- |
| `/usr/bin`   | Non-essential user binaries (`gcc`, `python`, `vim`)                 |
| `/usr/sbin`  | Non-essential system admin binaries                                  |
| `/usr/lib`   | Libraries for `/usr/bin` and `/usr/sbin`                             |
| `/usr/local` | Locally compiled/installed software (not managed by package manager) |
| `/usr/share` | Shared resources вЂ” docs, icons, man pages                            |

- рџ”ґ **Pentesting:** SUID bits on binaries in `/usr/bin`, vulnerable locally installed software in `/usr/local`.

---
### `/lib` and `/lib64` вЂ” Shared Libraries

- Shared libraries (`.so` files) required by binaries in `/bin` and `/sbin`.
- `/lib64` contains 64-bit libraries on 64-bit systems.
- рџ”ґ **Pentesting:** Outdated libraries can be exploited for privilege escalation via **library hijacking**.

---
### `/opt` вЂ” Optional / Third-Party Software

- Software installed outside the standard package manager.
- Examples: `/opt/google/chrome`, `/opt/metasploit`
- рџ”ґ **Pentesting:** Third-party software here is often less patched вЂ” look for vulnerable versions and misconfigurations.

---
### `/media` and `/mnt` вЂ” Mount Points

- `/media` вЂ” auto-mount point for removable media (USB, CD/DVD).
- `/mnt` вЂ” manual temporary mount point.
- рџ”ґ **Pentesting:** Mounted external media can be used for data exfiltration or malware delivery.

---
### `/boot` вЂ” Bootloader Files

| File         | Purpose                           |
| ------------ | --------------------------------- |
| `vmlinuz`    | Compressed Linux kernel image     |
| `initrd.img` | Initial RAM disk used during boot |
| `grub/`      | GRUB bootloader configuration     |

- рџ”ґ **Pentesting:** Tampering with GRUB config can establish **persistent access** or bypass authentication via single-user mode.

---
### `/srv` вЂ” Service Data

- Data served by system services (FTP, HTTP, etc.).
- Examples: `/srv/ftp`, `/srv/http`
- рџ”ґ **Pentesting:** Check for misconfigured service directories or weak permissions on hosted files.

---
### `/run` вЂ” Runtime Data

- Runtime state since last boot вЂ” PID files, sockets, lock files.
- Cleared on every reboot.
- рџ”ґ **Pentesting:** Socket files may be hijackable; sensitive runtime files may be left exposed.

---

### `/sys` вЂ” Kernel / Hardware Interface

- Virtual filesystem for interacting with hardware and kernel parameters at runtime.
- рџ”ґ **Pentesting:** Rarely targeted directly; useful for enumerating hardware and loaded kernel modules.

---
### `/lost+found`

- Present on every ext filesystem partition.
- Stores file fragments recovered during filesystem checks (`fsck`).
- рџ”ґ **Pentesting:** Recovered fragments may contain remnants of sensitive deleted files.

---

## Pentesting Quick Reference

| Directory       | Why It Matters                                                       |
| --------------- | -------------------------------------------------------------------- |
| `/etc`          | Credentials, misconfigs, `passwd`, `shadow`, SSH config, cron jobs   |
| `/var/log`      | Auth logs, brute-force traces, login history                         |
| `/tmp`          | World-writable вЂ” payload staging, exploit compilation                |
| `/home`         | SSH keys, `.bash_history`, personal data, backdoor `authorized_keys` |
| `/root`         | Full compromise вЂ” root's SSH keys, scripts, history                  |
| `/proc`         | Live process data вЂ” env vars and cmdline args leak credentials       |
| `/dev`          | Device access вЂ” keylogging via `/dev/input`                          |
| `/opt`          | Unpatched third-party software                                       |
| `/boot`         | Bootloader tampering for persistence                                 |
| `/lib` `/lib64` | Library hijacking for privilege escalation                           |
